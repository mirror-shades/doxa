    Typed Unions

    What are unions

    Unions are a special type which can represent one type or another. For instance, you can make a value that either returns a number, or a string. A union can only hold a single value, and therefore represent a single type when used.

    Syntax

    Unions are declared using the smiley face (: ... :) syntax:

    var x :: (: int, float :);
var y :: (: string, int, float :);

    The syntax consists of:

    • (: - Opening union delimiter
    • A comma-separated list of types
    • :) - Closing union delimiter

    Examples

    // A union that can be either an integer or a float
var number :: (: int, float :);
number is 42;     // Valid - integer
number is 3.14;   // Valid - float

// A union that can be a string, integer, or float
var value :: (: string, int, float :);
value is "hello"; // Valid - string
value is 100;     // Valid - integer
value is 2.718;   // Valid - float

// Unions default to the first type when not initialized
var defaulted :: (: int, float :);
defaulted?;       // Prints 0 (default int value)

var float_first :: (: float, int :);
float_first?;     // Prints 0.0 (default float value)

    Default Behavior

    When a union is declared without initialization, it defaults to the first type in the union with its default value:

    • int defaults to 0
    • float defaults to 0.0
    • string defaults to ""
    • tetra defaults to false
    • nothing defaults to nothing

    This ensures that unions always have a well-defined state and the compiler knows which type is currently active.

    Optional Values

    A common use case for unions is representing optional values using nothing:

    // Optional integer
var maybe_int :: (: int, nothing :);
maybe_int is 42;     // Has a value
maybe_int is nothing; // No value

// Function that might not return a value
fn findValue(arr :: array, target :: int) :: (: int, nothing :) {
    for (arr) |item| {
        if (item == target) {
            return item;
        }
    }
    return nothing; // Not found
}

    Why unions are useful

    Unions are particularly useful when:

    • A function can return different types depending on the input
    • A variable needs to hold different types at different times
    • You want to represent optional values (like (: T, nothing :))
    • You're working with data that can have multiple valid representations

    Type Safety Tiers

    Doxa provides three tiers of type safety for working with unions, from most dangerous to safest:

    1. Kill - Critical Assertions

    Use when you're confident about types and want fast failure. The program halts with a detailed stack trace if the type doesn't match.

    const intOrString :: (: int, string :) = getUserInput();

// Explicit type assertion - halt if not int
const num :: int is intOrString!int kill;

// Type inference - halt if type doesn't match context
const str is intOrString! kill;

    Use cases: Debugging, critical paths where type mismatches indicate bugs, performance-critical code.

    2. Catch - Graceful Recovery

    Use when you want to handle type mismatches gracefully. The catch block provides a fallback when the type doesn't match.

    const value :: (: int, string :) = getUserInput();

// Handle specific type mismatch
const num :: int is value!int catch {
    log("Expected int, got {}", typeof(value));
    return 0; // Provide default value
};

// Handle any type mismatch with inferred type
const processed = value! catch {
    log("Unexpected type: {}", typeof(value));
    return handleUnexpectedType(value);
};

    Key points:

    • The catch block executes only when the type doesn't match
    • You can return any value from the catch block
    • The catch block has access to the original union value

    3. Match - Exhaustive Safety

    Use for complete type safety and handling all possible cases. The as keyword provides a typed variable for each case.

    const result = addLimit(100, 200);

// Exhaustive matching with typed variables
match result as x {
    .int => {
        // x is typed as int here
        onlyUsesInts(x);
    },
    .errorList => {
        // x is typed as errorList here
        handleError(x);
    }
}

    Key points:

    • as provides a typed variable for each match arm
    • The variable is automatically typed to the matched variant
    • Exhaustive matching ensures all cases are handled
    • You can destructure complex unions with multiple variables

    Error Handling with Unions

    Unions combine with enums for expressive error handling without dedicated error types:

    // Define error types as enums
enum MathError {
    Overflow,
    Underflow,
    DivisionByZero,
}

// Function returns union of success or error
fn safeDivide(a :: int, b :: int) :: (: int, MathError :) {
    if (b == 0) return MathError.DivisionByZero;
    if (a > 255) return MathError.Overflow;
    if (a < 0) return MathError.Underflow;
    return a / b;
}

// Handle errors using any tier
const result = safeDivide(100, 0);

// Tier 1: Kill on error
const value :: int is result!int kill;

// Tier 2: Catch and handle
const value :: int is result!int catch {
    log("Math error occurred");
    return 0;
};

// Tier 3: Exhaustive handling
match result as x {
    .int => {
        processResult(x);
    },
    .MathError => {
        handleMathError(x);
    }
}

    Pattern Matching Details

    The as Keyword

    The as keyword in pattern matching provides typed variables:

    const union :: (: int, string, float :) = "hello";

// Single variable
match union as x {
    .int => processInt(x),      // x is typed as int
    .string => processString(x), // x is typed as string
    .float => processFloat(x),   // x is typed as float
}

    Exhaustiveness

    Match expressions must be exhaustive or include an else clause:

    const value :: (: int, string :) = "hello";

// Exhaustive - handles all cases
match value as x {
    .int => handleInt(x),
    .string => handleString(x),
}

// Non-exhaustive with else
match value as x {
    .int => handleInt(x),
    else => handleOther(x),
}

    Best Practices

    1. Prefer Match: Use exhaustive matching for maximum safety
    2. Use Catch for Recovery: When you need graceful fallbacks
    3. Reserve Kill for Debugging: Only use kill assertions in development or critical paths
    4. Be Explicit: Always specify the behavior you want rather than relying on defaults
    5. Handle All Cases: Use exhaustive matching when possible to catch all edge cases
